# 写作通用经验与陷阱

## 字数验证

**必须用Python正则统计中文字符，不能用字节数。**

UTF-8编码下，一个中文汉字占3字节。用`wc -c`或`os.path.getsize()`得到的字节数除以3才是大致字符数，但不精确（还有标点、英文等）。

正确方法：
```python
import re
with open(path, 'r', encoding='utf-8') as f:
    content = f.read()
chinese = len(re.findall(r'[\u4e00-\u9fff]', content))
```

排除元数据行（以`【`开头的行）后再统计。

**⚠️ 字数标注必须是真实统计结果，禁止AI估算。** 2026-07-05发现所有5本书的章节末尾字数标注都是AI写的假数字（如"约3100字"实际2708字，"约12500字"实际5819字，book5甚至有"XXXX"占位符）。每章写完后必须立即用Python统计并替换标注。

## 内容扩充策略

### 正确做法
1. write_file写入初稿（约1600-2000字核心情节）
2. Python统计字数
3. 未达标时用`terminal cat >> file << 'EOF'`追加内容（每次约1000-1500字）
4. 再次统计，重复直到达标
5. 最后5字以内时用patch微调

### 追加内容的方向（自然扩充，不显水字数）
- 战斗/训练场景的详细动作描写
- 角色之间的对话扩展（每人多说几句）
- 主角的心理活动和计划分析
- 环境描写和感官细节
- 具体过程的逐步展开

### 不要用patch扩充字数
patch需要精确匹配old_string，长文本匹配成本高，容易陷入死循环。每次patch成功只增加约200-400字，需要10+次才能补足6000字。

## 章节衔接与钩子

每章结尾必须有钩子，且与下一章直接相关。不能在章节末尾用空洞的心灵鸡汤段落代替真正的剧情钩子。

**禁止的结尾模式（模板化）：**
- 同一角色的内心独白反复出现（如"李玄的心中充满了信心"）
- 同一句口号反复出现（如"李家，从今天起，将由他来守护"）
- 同一感慨反复出现（如"修仙世界……也许这就是我的命运吧"）
- 同一动作反复出现（如"他的嘴角微微上扬，露出一丝微笑"）
- 同一结尾语反复出现（如"而这条路，才刚刚开始"）

这些段落每章最多出现一次（通常是第1章）。后续章节必须用具体的剧情悬念作为结尾钩子。

## 每章必须独立

每次cronjob或写作任务只写1章。不要试图在一个任务里写多章。每章有独立的上下文，质量更稳定，不易偏离大纲。

## 爽点写作指南（签约核心！）

**番茄是免费阅读平台，编辑最看重"爽点密度"。** "增加爽点"不能只是一句口号，Writer Agent需要具体的可执行技法。以下是经过验证的爽点模式：

### 爽点的本质：预期违背 + 即时反馈

读者爽的不是"主角赢了"，而是"主角被看不起→用实力打脸→旁人被震住"这个**节奏链**。缺少任何一个环节，爽感就打折。

### 必须掌握的5个爽点模式

#### 模式1：电话反击（最适合中前期）
```
反派/配角打电话来嘲讽或施压 → 主角不解释、不慌张 → 轻描淡写一句 → 对方态度180度转变
关键：主角越平静，读者越爽。旁听者震惊反应是爽点放大器。
示例骨架：
"你算什么东西也配跟我谈条件？"对方在电话那头冷笑。
主角没说话，只说了一个名字。
电话那头沉默了三秒。
"您、您是……"
旁边一直瞧不起主角的人，手里的茶杯停了。
```

#### 模式2：能力展示（适合训练/战斗/专业场景）
```
先让对手轻视 → 主角出手 → 结果超出所有人预期 → 旁观者反应
关键：展示要具体，不能写"他很强"。要写动作细节、速度、力度、旁人感受。
错误写法："李玄一拳打出，力量惊人，对手被打飞。"
正确写法：拳头落在对方胸口的那一刻，对方的眼睛瞪大了——不是疼，是来不及躲。
```

#### 模式3：信息差碾压（适合智斗/商战）
```
反派以为掌握了优势 → 主角其实早就知道/准备了后手 → 反转
关键：提前埋线索，反转时读者回头看会恍然大悟。不要凭空反转。
```

#### 模式4：身份揭露（适合打脸）
```
反派当众羞辱主角 → 有人认出主角真实身份 → 反派脸色变化 → 主角淡淡回应
关键：揭露要由第三方说出来，不能主角自己炫耀。旁人的震惊比主角的宣言更有说服力。
```

#### 模式5：小胜积累（适合日常/经营线）
```
主角做了一件小事 → 结果带来意外收益 → 配角发现异常 → 追问下更大布局
关键：每章一个小爽点，不用每次都大场面。攒钱、做生意、修炼突破都可以是小爽点。
```

### 爽点分布节奏

- **前3章**：至少每章一个明确爽点。第1章可以是"金手指觉醒"类大爽点，第2-3章用能力展示或小胜积累维持。
- **第4-10章**：每2-3章一个大爽点，中间穿插小爽点（旁衬、小胜利）。
- **第10-20章**：插入一个"电话反击"或"身份揭露"作为中段高潮，防止节奏疲软。
- **每章结尾**：可以是一个小悬念钩子，不一定是大爽点。但也不能完全没有期待感。

### 爽点常见失败模式

| 失败 | 原因 | 修正 |
|------|------|------|
| 主角赢了但读者无感 | 没有"被看不起"的前置铺垫 | 加一段反派/配角的轻视或嘲讽 |
| 爽点太突然 | 缺乏铺垫和线索 | 提前2-3章埋暗示 |
| 旁衬缺失 | 只有主角反应，没有观众反应 | 加1-2个旁观者的震惊/后悔描写 |
| 对话像说明书 | 爽点靠旁白说而不是场景呈现 | 改成动作+对话+旁人反应 |
| 一章多个大爽点 | 节奏爆炸，后面乏力 | 每章聚焦1个核心爽点，其他做铺垫 |

### 开篇3章特别要求

第1-3章是签约审核的重中之重。每章必须有明确的爽点设计：

- **第1章**：金手指觉醒 / 重生确认 / 核心矛盾爆发。结尾留大悬念。
- **第2章**：主角第一次展示优势（哪怕很小）。让读者看到"这本书有意思"。
- **第3章**：第一个小高潮——打脸/逆袭/意外收获。证明主角不是普通人。

如果第3章评估时"爽点密度"低于6分，**暂停写作，优先修改前3章**，不要继续往下写。

## 写作前校验大纲（强制步骤）

每次写作任务开始前，必须：
1. 读取outline.md，确认当前章节号落在哪个分卷区间
2. 读取该分卷对应的关键事件描述
3. 读取最近2章的实际内容，确认剧情走向
4. 如果大纲要求与已有章节不符，先纠正方向再继续写

## 卷完结后一致性检查

每卷写完30章后，必须：
1. 提取每章标题和关键实体
2. 逐条对照outline中的"关键事件"描述
3. 发现脱节的，用实际章节内容重写outline
4. 同步更新characters.md
5. 更新write_chapter.sh的prompt

## 技术注意事项

- write_file单次输出约1600-2000字中文，超过会截断
- delegate_task的write_file参数不能超过约8K tokens，否则stream超时
- 章节文件命名统一为`chNNN_第X章 标题.md`
- bash八进制陷阱：`ch008`中的`008`会被bash当作八进制报错，需要用`$((10#$num))`转换
- publish_fanqie.py使用`CURRENT_VOLUME_ID`和`CURRENT_VOLUME_NAME`，不是`VOLUME_ID`/`VOLUME_NAME`
- config.env统一标准：MIN_WORDS=2000, MAX_WORDS=3500, CHAPTER_WORDS_TARGET=2500（2026-07-06更新为番茄平台标准字数）
- **delegate_task的goal参数中字数要求必须从config.env动态读取**，不能硬编码。修改config.env后必须检查所有delegate_task调用中的goal是否同步更新。详见 `references/word-count-standard.md`。
